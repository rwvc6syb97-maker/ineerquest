## 1. 全局强制约定

### 1.1 统一返回结构（不可变更）

```ts
interface ApiResponse<T = unknown> {
  code: number;      // 业务码，见 §2
  message: string;   // 人类可读提示
  data: T;           // 业务数据；无数据时为 null
}
```

- **HTTP 状态码**：与业务码语义同步（200/400/401/403/404/409/429/500）。
- **列表 data 结构**：`{ list: T[]; total: number; page: number; pageSize: number }`。
- **成功**：`code=200`；`4xxx`=客户端错误；`5xxx`=服务端异常；`9xxx`=限流/系统。

### 1.2 code 设计规则

| 段 | 含义 |
|----|------|
| `200` | 成功 |
| `40xx` | 通用/鉴权客户端错误 |
| `41xx` | 用户账户 |
| `42xx` | 测评 |
| `43xx` | 报告 |
| `44xx` | 职业推荐 |
| `45xx` | AI 对话 |
| `46xx` | 激活码/会员 |
| `47xx` | 辅导预约 |
| `48xx` | 后台管理 |
| `50xx` | 服务端异常 |
| `90xx` | 限流/系统繁忙 |

### 1.3 命名规范（三端强制一致）

- **接口字段 / 前端**：`camelCase`（驼峰）。
- **数据库字段**：`snake_case`（下划线），由 Prisma `@map` 做映射转换。
- **路由路径**：小写短横线，资源名复数；C 端前缀 `/api/v1`，后台前缀 `/api/v1/admin`。
- **示例映射**：接口 `userId` ↔ DB `user_id`；接口 `createdAt` ↔ DB `created_at`。

### 1.4 时间规范（权威裁定 v2.2，统一 ISO8601 UTC）

- 后端内部与 DB 统一存 **UTC**（`DATETIME`）。
- **接口返回统一为 ISO8601 UTC 字符串**（带 `Z` 后缀）：`YYYY-MM-DDTHH:mm:ss.sssZ`，如 `2026-07-10T03:20:00.000Z`。
- **弃用**旧约定的"北京时间字符串 `YYYY-MM-DD HH:mm:ss`"作为接口返回格式。
- **前端负责本地化展示**：由前端统一 util 将 ISO8601 UTC 转东八区展示，不再依赖后端下发本地化字符串。
- **修复归属（P2 可迭代）**：
  - 后端：报告 `createdAt` 等原返回北京时间字符串的字段，统一改为 ISO8601 UTC；测评 `startedAt`/`completedAt`、支付 `expireAt` 已为 UTC，保持 `.sssZ` 完整格式。
  - 前端：所有时间字段接入统一 `formatDateTime(utcStr)` 做时区换算展示。

### 1.5 鉴权与请求头

- `Authorization: Bearer <accessToken>`。
- **双 Token**：`accessToken` 有效期 30 分钟，`refreshToken` 30 天。
- **黑名单**：登出 / 封禁后写入 MySQL `cache_kv` 表（`key=blacklist:{jti}`，TTL 由 `expire_at` 控制），无 Redis。
- `X-Trace-Id`（可选）全链路追踪。

### 1.6 限流策略（MySQL 令牌桶）

- 全局：**100 次/分钟/用户**，超限返回 `code=9001` + HTTP 429 + `Retry-After` 头。
- 计数落在 MySQL `cache_kv` / `cache_zset` 表，滑动窗口按 `key=rate:{scope}:{id}`。
- 特殊接口单独更严格：短信验证码 1 次/60s；登录 5 次/分；AI 对话按日配额。

### 1.7 数据存储替代 Redis 约定

| 原 Redis 用途 | MySQL 替代方案 |
|--------------|---------------|
| 键值缓存 / JWT 黑名单 | `cache_kv(key, value, expire_at)` |
| 排队 / 令牌桶 / 限流窗口 | `cache_zset(key, member, score)` |
| 答题草稿断点续答 | `assessment_progress` 表 |
| 分布式锁（辅导时段） | MySQL 行锁 `SELECT ... FOR UPDATE` + 唯一约束 |

### 1.8 幂等约定

- 所有写接口支持 `Idempotency-Key` 请求头。
- 激活码兑换按 `code_no` 唯一约束幂等；辅导下单按 `(userId, scheduleId)` 唯一约束幂等。

## 3. 通用异常场景矩阵

> 每个接口在实现与测试时**必须**覆盖以下边界，除非明确标注不适用。

| 场景 | 预期行为 | 统一码 |
|------|---------|--------|
| 空输入 | 校验拦截，提示具体缺失字段 | 4000 |
| 格式错误 | 手机号/邮箱/枚举值不合法即拦截 | 4000 |
| 参数超长 | 超字段上限拦截 | 4001 |
| 重复提交 | 幂等键或唯一约束返回上次结果或冲突 | 4090 |
| 登录凭证过期 | 401，前端触发 refresh；refresh 也失效则跳登录 | 4010/4012 |
| 越权访问 | 校验资源归属/角色，拒绝 | 4030 |
| 第三方失败 | 捕获降级，不透传上游堆栈 | 5002 |
| 上游超时 | 设置超时阈值，超时即返回 | 5003 |

| 并发请求 | 行锁/唯一约束/乐观锁保证一致 | 4090 |
| 频繁调用限流 | 令牌桶拦截 | 9001 |
| 软删除逻辑 | `is_deleted=1` 记录对 C 端不可见 | 4040 |
| 权限隔离 | 普通用户/管理员接口物理隔离，`/admin/*` 需管理员 | 4030 |

## 4. 模块 1：用户账户

**功能**：手机验证码 + 邮箱密码双通道登录、双 Token 刷新、黑名单登出、资料/隐私管理、注销冷静期。

### 4.1 接口清单

| # | 方法 | 路由 | 说明 | 鉴权 | 限流 |
|---|------|------|------|------|------|
| 1 | POST | `/api/v1/auth/sms/send` | 发送短信验证码 | 否 | 1 次/60s/手机号 |
| 2 | POST | `/api/v1/auth/login/sms` | 手机验证码登录（无则注册） | 否 | 5 次/分 |
| 3 | POST | `/api/v1/auth/login/email` | 邮箱密码登录 | 否 | 5 次/分 |
| 4 | POST | `/api/v1/auth/register/email` | 邮箱注册 | 否 | 3 次/分 |
| 5 | POST | `/api/v1/auth/refresh` | 刷新 Token | refreshToken | - |
| 6 | POST | `/api/v1/auth/logout` | 登出（加黑名单） | 是 | - |
| 7 | GET | `/api/v1/users/me` | 当前用户资料 | 是 | - |
| 8 | PATCH | `/api/v1/users/me` | 更新资料 | 是 | - |
| 9 | GET | `/api/v1/users/me/privacy` | 获取隐私设置 | 是 | - |
| 10 | PATCH | `/api/v1/users/me/privacy` | 更新隐私设置 | 是 | - |
| 11 | POST | `/api/v1/users/me/deactivate` | 申请注销（30 天冷静期） | 是 | - |

### 4.2 入参 / 出参

**① 发送验证码** `POST /auth/sms/send`
- 入参：`phone` string 必填（大陆手机号正则 `^1[3-9]\d{9}$`）；`scene` string 必填（`login`/`bind`）。
- 出参：`{ expireIn: number }`（验证码有效秒数，默认 300）。

**② 手机验证码登录** `POST /auth/login/sms`
- 入参：`phone` string 必填；`code` string 必填（6 位数字）。
- 出参：`{ accessToken, refreshToken, user: UserProfile, isNewUser: boolean }`。

**③ 邮箱密码登录** `POST /auth/login/email`
- 入参：`email` string 必填（邮箱格式）；`password` string 必填（8~32 位）。
- 出参：同 ②（无 `isNewUser`）。

**④ 邮箱注册** `POST /auth/register/email`
- 入参：`email` 必填；`password` 必填（8~32，含字母+数字）；`nickname` 可选（≤20）。
- 出参：`{ userId, email }`。

**⑤ 刷新 Token** `POST /auth/refresh`
- 入参：`refreshToken` string 必填。
- 出参：`{ accessToken, refreshToken }`（轮换刷新）。

**⑦ UserProfile 结构**（`GET /users/me` 的 data）
```ts
{ userId: number; userNo: string; nickname: string; avatar: string;
  phone: string;      // 脱敏 138****8000
  email: string;      // 脱敏
  role: number;       // 1 普通 / 2 辅导师 / 3 管理员
  membershipLevel: number; // 0 免费 / 1 Pro / 2 辅导
  membershipExpireAt: string | null;
  createdAt: string; }
```

### 4.3 专属错误码（41xx）

| code | message | 触发条件 |
|------|---------|---------|
| 4101 | 验证码错误或已过期 | code 不匹配/超时 |
| 4102 | 验证码发送过于频繁 | 60s 内重复发送 |
| 4103 | 账号或密码错误 | 邮箱登录凭证不符 |
| 4104 | 邮箱已注册 | 注册时唯一约束冲突 |
| 4105 | 账号已被封禁 | `user.status=2` |
| 4106 | 账号处于注销冷静期 | 已申请注销未撤销 |
| 4107 | 密码强度不足 | 不满足 8~32 位含字母数字 |

| 用例 | 输入 | 预期 |

## 5. 模块 2：MBTI 测评

**功能**：40 题（4 页×10 题）→ 计算 16 型人格 → 生成结果；支持断点续答。题库按 `question_version` 隔离。

### 5.1 接口清单

| # | 方法 | 路由 | 说明 | 鉴权 |
|---|------|------|------|------|
| 1 | GET | `/api/v1/assessments/questions` | 拉取当前版本题库（40 题） | 是 |
| 2 | POST | `/api/v1/assessments/records` | 创建测评记录（开始） | 是 |
| 3 | PATCH | `/api/v1/assessments/records/:id/answers` | 分页暂存答案（断点续答） | 是 |
| 4 | POST | `/api/v1/assessments/records/:id/submit` | 提交答卷，计算 MBTI 类型 | 是 |
| 5 | GET | `/api/v1/assessments/records/:id/result` | 获取测评结果 | 是 |
| 6 | GET | `/api/v1/assessments/records` | 我的测评历史（分页） | 是 |
| 7 | GET | `/api/v1/assessments/records/latest` | 获取最近未完成记录（续答入口） | 是 |

### 5.2 入参 / 出参

**① 题库** `GET /api/v1/assessments/questions` → data（权威裁定 v2.2，三方以此为准，字段名/类型/结构不得私改）：

```ts
interface QuestionBank {
  /** 题库版本号，如 'v1' */
  version: string;              // 必填
  /** 题目总数（如 40），供前端 length/分页计算 */
  total: number;               // 必填
  /** 扁平题目数组，已按 dimension→sortOrder 全局排序；前端直接 slice 分页 */
  questions: Array<{
    /** 题目主键，字符串（BigInt 序列化，避免精度丢失） */
    id: string;                // 必填
    /** 维度字符串（后端将内部 number 编码映射为字符串下发） */
    dimension: 'EI' | 'SN' | 'TF' | 'JP';  // 必填
    /** 题干文案 */
    content: string;           // 必填
    /** 全局排序序号，供稳定分页 */
    sortOrder: number;         // 必填
    /** 是否反向计分题（后端计分用，前端仅透传） */
    isReverse: boolean;        // 必填
    /** 选项数组，已按 sortOrder 排序 */
    options: Array<{
      /** 选项主键，字符串；作答提交时作为 optionId 上送 */
      id: string;              // 必填
      /** 选项标识键，如 'A'/'B'（后端计分辅助） */
      optionKey: string;       // 必填
      /** 选项展示文案（前端直接展示，等价于旧 label） */
      content: string;         // 必填
      /** 极性，如 'E'/'I'（后端计分用，前端不用） */
      polarity: string;        // 必填
      /** 该选项得分（后端计分用，前端不用） */
      score: number;           // 必填
      /** 选项排序序号 */
      sortOrder: number;       // 必填
    }>;
  }>;                          // 必填
}
```

**裁定说明（三方对齐指令，P0 阻断上线）：**
- **结构裁定：扁平数组**。`questions` 为全局排序后的扁平 `Array`，**弃用**后端按维度分组的 `dimensions: {EI:[],SN:[],TF:[],JP:[]}` 对象结构。理由：QuizPage 依赖 `questions.slice()`/`questions.length` 分页答题，扁平顺序是核心链路刚需。
- **dimension 字段裁定：字符串** `'EI'|'SN'|'TF'|'JP'`，**弃用** number 编码对外暴露。后端在组装时按 `DimensionKey` 将内部 number 映射为字符串下发。前端据此取维度主题色与展示。
- **option 命名裁定**：权威字段为 `{ id, optionKey, content, polarity, score, sortOrder }`。**弃用**前端 `{ value, label }`。展示文案统一用 `content`（等价旧 label）；作答提交时上送 `optionId = option.id`。`polarity`/`score`/`optionKey` 由后端计分使用，前端仅存在不消费（本地评分已随 T2 移除）。
- **修复归属**：
  - **后端**：将 `getQuestions` 返回结构由分组对象 `{version,total,dimensions}` 改为扁平 `{version,total,questions:[]}`；`dimension` number→字符串映射后下发。
  - **前端**：`Question` 类型改为 `{ id, dimension, content, sortOrder, isReverse, options:[{id,optionKey,content,polarity,score,sortOrder}] }`；`getQuestions` 返回类型由 `Question[]` 改为 `QuestionBank`，取用 `resp.questions`；QuizPage 将 `opt.label`→`opt.content`，移除 `opt.value` 依赖。

**② 创建记录** 入参：`questionVersion` string 必填。出参：`{ recordId: number; recordNo: string; startedAt: string }`。

**③ 暂存答案** 入参：
- `page` number 必填（1~4）；`answers` 数组必填，元素 `{ questionId: number; optionId: number }`。
- 出参：`{ savedCount: number; answeredTotal: number }`。

**④ 提交** 出参：
```ts
{ recordId: number; mbtiType: string; // 如 'ENFJ'
  dimensionScores: { EI: number; SN: number; TF: number; JP: number };
  resultId: number; reportId: number | null }
```
> **reportId 取值裁定（见 §13.2 B8）**：提交④**正常成功路径下 `reportId` 恒为非空 number**（因 B1 事务内同步建 report）。类型标注保留 `number | null` 仅为兼容"历史遗留记录读取/极端并发回读兜底"场景；前端提交成功分支可直接用 reportId 跳转，无需为 null 编写重试逻辑。若建 report 失败，后端须回滚事务并返 5xxx，即"提交失败可重试"，而非返回 reportId=null。

```ts
  /** 摘要文案，纯字符串（后端由结构化 summary 渲染成文案下发，前端直接展示） */
  summary: string;            // 必填
  /** 四维度得分，供雷达图；score 为 0-100 或后端约定区间 */
  dimensions: {
    dimension: 'EI' | 'SN' | 'TF' | 'JP';
    left: string;             // 左极标签，如 '内向 I'
    right: string;            // 右极标签，如 '外向 E'
    score: number;
  }[];                        // 必填，固定 4 项
  /** 是否已解锁付费深度内容 */
  isUnlocked: boolean;        // 必填
  /** 章节对象数组：未解锁时仅含免费预览章节，付费章节内容不下发 */
  sections: {
    sectionKey: string;
    title: string;
    content: object;          // 结构化内容对象
    sortOrder: number;
    paid: boolean;
  }[];                        // 必填（可为空数组）
  /** 未解锁的付费章节 key 列表；已解锁时为 [] */
  lockedSectionKeys: string[];  // 必填
  /** 深度生成状态全集（后端将内部数字枚举映射为下列字符串） */
  generateStatus: 'pending' | 'generating' | 'done' | 'failed';  // 必填
  /** 创建时间，ISO8601 UTC 字符串（见 1.4 时间规范；前端本地化展示） */
  createdAt: string;          // 必填

```

**裁定说明（三方对齐指令）：**
- **弃用** `previewSections`/`lockedSections` 双 key 数组；概览统一返回 `sections`（对象数组，含预览内容）+ `lockedSectionKeys`（string[]）。
- **状态字段**统一用 `generateStatus`（字符串枚举），**弃用**后端顶层 `status`（数字枚举）对外暴露。映射规则：内部 `GENERATING(0)→'generating'`、`READY(1)→'done'`、`FAILED(2)→'failed'`；报告已创建但深度内容尚未触发生成时下发 `'pending'`。报告记录物理状态（可为 READY）与对外 `generateStatus` **解耦**：初始建报告即写入免费预览章节、`isUnlocked=0`，但因深度三段未触发 LLM，对外 `generateStatus` 仍为 `'pending'`——**免费预览可看 ≠ 深度已生成**，两者不冲突（详见 §13.2 B7）。
- 后端需在概览接口顶层**新增组装** `recordId`、`family`（由 mbtiType 推导）、`summary`（渲染为 string）、`dimensions`（取自 dimension_scores 章节四维得分）。前端**不得**再从 sections 内部反解这些字段。
- 前端 `generateStatus` 类型需补齐 `'pending'`，并去除对 sections 反解 family/dimensions 的逻辑。
